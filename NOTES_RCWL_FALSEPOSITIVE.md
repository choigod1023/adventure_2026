# RCWL-0516 차량/보행자 구분 문제

> **현재 구현 상태**: `evaluateSensor()` 는 **PIR 과 RCWL 이 동시에 HIGH(AND) 면 위험**으로 판정.
> 레벨 기반이라 어느 쪽이 먼저 떴든 둘 다 HIGH 인 순간 트리거(01→11, 10→11 모두 동작).
> 오감지를 더 줄이려면 방안 ①(PIR 게이팅)/②(차폐)/⑤(거리 축소) 참고.

## 문제 정의

RCWL-0516은 **마이크로파 도플러 레이다** = "움직임이 있다/없다"의 디지털 신호만 출력. 차량인지 사람인지 **본질적으로 구분 불가**.

회의록의 설계 ("차량과 보행자 **동시** 감지 시 위험")는 RCWL이 차량 전용으로 작동한다는 전제인데, 실제로는:
- 차량 통과 → RCWL HIGH ✅
- 사용자(스몸비) 본인이 움직임 → RCWL HIGH ❌ (오감지)
- 옆 사람 지나감 → RCWL HIGH ❌ (오감지)

## RCWL 센서 특성 (참고)

| 항목 | 값 |
|---|---|
| 방식 | 3.18GHz 마이크로파 도플러 |
| 감지거리 | 5~7m (조정 가능, CDS 저항) |
| 출력 | 디지털 HIGH (2-3초 유지) |
| **벽 투과** | **OK** (얇은 벽/플라스틱 통과) ← 방향성 문제의 원인 |
| 속도 정보 | ❌ 없음 |

## 해결 방안 5가지

### ① PIR 게이팅 (소프트웨어, 최우선) ⭐⭐⭐

**핵심 아이디어**: PIR(HC-SR505)은 **인체 적외선**만 감지 → 사람만 구별 가능.

```
규칙:
- PIR HIGH + RCWL HIGH 동시  →  "사람 본인 움직임" 가능성 → 무시
- PIR LOW  + RCWL HIGH       →  "비-사람 움직임" → 차량 가능성 ⭐
- PIR HIGH 안정 + RCWL pulse →  "사용자 정지 중 외부 움직임" → 차량 가능성 ⭐
```

**시간 윈도우 적용**:
```cpp
// 사용자가 도착해서 신호등 앞에 섰음
// PIR이 처음 HIGH → 그 후 사람이 정지하면 PIR LOW (HC-SR505는 정지 시 풀림)
// 이 상태에서 RCWL HIGH가 뜨면 차량으로 간주

if (PIR_was_high_within(3000ms) && PIR_now_low && RCWL_high) {
    // 사용자가 앞서 도착해 정지 중 + 외부 움직임 → 차량 확정도↑
    danger = true;
}
```

**장점**: 코드만으로 해결, 추가 부품 X
**단점**: 사용자가 움직이는 순간 사각지대 발생

---

### ② 물리적 차폐 + 지향성 (하드웨어) ⭐⭐⭐

RCWL의 PCB 안테나는 보드 한쪽에 있음. 반대편을 **알루미늄 호일/구리테이프**로 덮으면 사용자 방향 신호 차단.

```
        도로 ↓
   ┌──────────────┐
   │  RCWL 안테나 │   ← 감지 방향
   ├──────────────┤
   │ ░░░ 차폐 ░░░ │   ← 알루미늄/구리테이프
   └──────────────┘
        ↑ 사용자 (스몸비)
```

회의록에 "**구리테이프**"가 예산 항목에 있는 게 우연이 아님 — 차폐 용도로 활용 가능.

**장점**: 매우 효과적, 단순
**단점**: 설치 방향 잘못되면 작동 안 함

---

### ③ C-ITS SPAT 교차 검증 (API, 보조) ⭐⭐

신호등 상태 = 차량이 지금 움직여야 하는지 안 움직여야 하는지의 정답.

```cpp
float carGreen = item["{방향}StsgRmdrCs"];   // 차량 직진 잔여
float pedGreen = item["{방향}PdsgRmdrCs"];   // 보행자 잔여

if (carGreen > 0 && carGreen < 36001) {
    // 지금 차량 신호 청색 = 차량이 진입 중일 가능성
    // RCWL HIGH 신뢰도↑
}
```

**장점**: API 모드에서 매우 강력
**단점**: API 모드에서만 동작, 네트워크 의존

---

### ④ 트리거 패턴 분석 (소프트웨어) ⭐

차량과 보행자의 RCWL 트리거 양상이 다를 수 있음:

| 객체 | 트리거 양상 |
|---|---|
| 차량 통과 | 짧고 강한 1회 펄스 (1-3초) |
| 보행자 | 산발적 짧은 펄스 반복 |
| 정지된 사용자 | 거의 트리거 안 됨 |

```cpp
// 최근 10초 내 트리거 횟수가 1~2회 = 차량 가능성
// 5회 이상 = 보행자 가능성
```

**장점**: 추가 비용 0
**단점**: 신뢰도 낮음, 튜닝 필요

---

### ⑤ RCWL 감지거리 축소 ⭐

RCWL의 C-TM 핀(또는 R-GN 저항)으로 감지 범위를 **5m → 2m로 좁히면** 사용자 본인이 들어오는 범위가 작아지고, 도로 쪽 차량만 잡힐 확률↑

기본 5m 범위는 너무 넓어서 사용자도 잡힘. 줄이는 게 베이스라인.

**장점**: 단순
**단점**: 차량도 가까이 와야 잡히므로 경고 시간 줄어듦 (트레이드오프)

---

## 추천 조합

**1차 (소프트웨어만)**: ① PIR 게이팅 + ⑤ 감지거리 2-3m로 축소
**2차 (케이스 제작 시)**: ② 구리테이프 차폐 추가
**3차 (API 모드)**: ③ C-ITS 교차 검증

### 통합 의사코드

```cpp
struct DetectionState {
  bool pir_now;
  bool rcwl_now;
  unsigned long pir_last_high_ms;
  unsigned long rcwl_last_high_ms;
};

bool isVehicleDetected(DetectionState &s) {
  // 케이스 A: PIR이 최근 활성 → 사용자 정지 + RCWL = 차량
  bool userArrivedRecently = (millis() - s.pir_last_high_ms) < 5000;
  if (userArrivedRecently && !s.pir_now && s.rcwl_now) {
    return true;
  }

  // 케이스 B: PIR + RCWL 동시 → 본인 움직임 가능성 → 보류
  if (s.pir_now && s.rcwl_now) {
    return false;
  }

  // 케이스 C: PIR 전혀 없는데 RCWL → 외부 객체 → 차량 가능성
  if (!userArrivedRecently && s.rcwl_now) {
    return true;
  }

  return false;
}
```

회의록의 원래 정의 ("차량 AND 보행자 동시 = 위험")는:
- 보행자 감지 = "사용자가 이 위치에 도착했음" (PIR이 최근에 한 번이라도 HIGH)
- 차량 감지 = 위 `isVehicleDetected()` 결과

이렇게 분리해서 해석하면 일관성 유지됨.
